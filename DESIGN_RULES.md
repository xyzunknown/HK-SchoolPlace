# Design Rules

## 1. 目的

本文件用于约束后续 AI 或开发者在实现功能时的 UI 变更范围，避免在开发核心业务时顺手把视觉系统带偏。

适用场景：

- 功能开发
- 页面补全
- 后台页面新增
- 组件改造

不适用场景：

- 全站品牌升级
- 全量视觉重构
- 新版设计系统从零重做

## 2. 设计结论

当前阶段需要：

- 先冻结设计方向
- 允许小范围组件演进
- 不做全站重设计

一句话原则：

**功能优先，视觉延续，局部精修，不做推翻式重做。**

## 3. 当前视觉方向

### 3.1 总体风格

- 延续当前绿色主色系
- 延续当前轻玻璃态 `glassmorphism` 风格
- 延续“vacancy-first” 信息层级
- 用户端保持轻盈、透明、偏消费产品风格
- 后台保持更克制、更高信息密度的管理风格

### 3.2 不要改变的核心印象

- 首页和用户端页面的绿色品牌识别
- 玻璃卡片、柔和阴影、圆角体系
- vacancy 状态作为最显眼信息
- 当前中英混排与港式繁体中文语气

### 3.3 允许微调的范围

- 间距、密度、对齐
- 按钮状态
- 表格可读性
- 空态与错误态文案
- 登录弹层交互细节
- 后台管理页布局补全

## 4. 设计优先级

设计决策优先级从高到低：

1. 信息是否清晰
2. vacancy 状态是否足够显眼
3. 移动端是否可用
4. 交互是否一致
5. 视觉是否精致

说明：

- 只要影响前 3 项，就不要为了“更好看”牺牲它们。

## 5. 当前应冻结的设计边界

### 5.1 颜色

- 主色继续使用绿色系
- vacancy 状态色继续保持：
  - available：绿色
  - waiting：橙色
  - full：灰色
  - unknown：浅灰
- 不引入新的主品牌色
- 不切成深色主题作为默认主题

### 5.2 圆角与卡片

- 延续大圆角卡片
- 延续轻玻璃背景与柔和边框
- 不改成强烈拟物风
- 不改成极简纯线框风

### 5.3 字体与语言

- 保持当前中文优先、繁体中文界面
- 保持港式用语与产品语气
- 不在局部页面改成简体中文
- 不混入明显不同语气的大陆式后台文案

### 5.4 页面结构

- 首页保留 Hero + 学段入口结构
- 列表页保留 搜索 + 筛选 + 排序 + 卡片列表 结构
- 详情页保留“学位空缺优先、学校资料其次”的结构
- 后台保留总览 / 列表 / 日志的管理模式

## 6. 组件级约束

### 6.1 必须尽量复用的组件

- `SchoolCard`
- `VacancyBadge`
- `FilterChips`
- `SearchBar`
- `SortSelect`
- `EmptyState`
- `LoginModal`
- `Navbar`

原则：

- 新功能先复用已有组件
- 只有在现有组件无法承载需求时，才新增组件
- 新增组件应优先继承现有 token、圆角、阴影、按钮风格

### 6.2 学校卡片

- 卡片必须继续突出 vacancy 状态与数量
- 学校名称、地区、类型标签继续保留现有层级
- 收藏、对比、查看详情按钮位置应保持稳定
- 不要把卡片改成信息过密的后台式表格块

### 6.3 登录弹层

- 可以改真实 Supabase 登录逻辑
- 但交互形态继续保持弹层，不改整页跳转式登录页，除非明确决策变更
- 关闭、登录成功返回原操作，这条体验应保留

### 6.4 筛选组件

- 保持 chip 形式
- 不在 MVP 阶段改成复杂抽屉筛选器，除非移动端已明显不可用
- 筛选项数量增加时，优先做折叠与分组，不先推翻现有交互

### 6.5 后台表格

- 后台以效率优先
- 允许更高信息密度
- 允许降低玻璃感、增强边界感
- 但不要做成与用户端完全断裂的另一套系统

## 7. 现在必须先拍板的设计项

这些项目在进入下一轮开发前，方向要固定：

1. 登录弹层在接入真实 Auth 后是否仍维持现有视觉与交互
2. 收藏 / 对比按钮的默认、激活、加载、失败状态样式
3. 后台 Alias 管理页的信息结构
4. 后台 unmatched record 页的信息结构
5. 管理员处理 unmatched 时的关键交互文案

说明：

- 上述项目只需定结构和交互，不需要先出高保真稿。

## 8. 现在不要做的设计工作

- 不重做首页品牌风格
- 不重做全站导航
- 不重做 design token
- 不引入完整新 design system
- 不做暗黑模式专项
- 不做地图主视图
- 不为后台单独发明一整套新视觉语言

## 9. 页面设计建议

### 9.1 用户端页面

- 继续强调“先看到有没有学位”
- 所有关键信息优先级：
  - vacancy 状态
  - vacancy 更新时间
  - 学校名称
  - 地区 / 类型 / 时段
- 移动端优先保证卡片可扫读、按钮可点击

### 9.2 后台页面

- 优先让管理员“能快速判断、快速处理”
- 重点不是漂亮，而是：
  - 筛选快
  - 状态清楚
  - 操作明确
  - 错误可追踪
- 对后台而言，信息密度可以高于用户端

## 10. AI 开发时的设计约束

给其他 AI 的默认设计限制：

1. 默认保持当前视觉方向，不要擅自重设计。
2. 做功能时优先复用现有组件和 token。
3. 如果需要新增页面，先沿用现有版式、按钮、卡片、chip 风格。
4. 如果认为现有设计不够用，先做最小增量修改，不要一次性推翻。
5. 若某项设计变更会影响多个页面，必须先更新本文件再实施。

## 11. 推荐的设计工作顺序

1. 先固定交互边界
2. 再做功能开发
3. 功能闭环后再做局部视觉打磨
4. 最后再考虑统一优化首页、品牌感、动效等非核心项

## 12. 当前结论

现在需要的是：

- 设计规则定下来
- 关键交互结构定下来
- 功能开发继续推进

现在不需要的是：

- 全站 UI 重做
- 视觉风格重新选型
- 先做完整高保真设计稿再开发
