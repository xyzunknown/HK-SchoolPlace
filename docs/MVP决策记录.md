# HKSchoolPlace MVP 决策记录

本文档记录 MVP 开发前需要拍板的所有关键决策，避免开发过程中反复讨论。

## 决策原则

- MVP 就是"最小可行产品"——能验证核心假设即可
- 优先降低开发和数据复杂度
- 能后补的功能不在第一版硬塞

---

## D1：MVP 先做哪些学段？

**决策：MVP 第一版仅做幼稚园（kg）**

理由：
- 幼稚园的 vacancy 数据源最明确（EDB 官方页面）
- 一个学段跑通完整闭环（抓取→匹配→展示→收藏→对比）再拓展
- 小学、中学的筛选逻辑不同（校网、Band），同时做会显著增加前后端工作量
- 产品验证的核心假设是"家长愿意用这个工具查学位"，一个学段足够验证

影响：
- 前端学段切换器保留，但小学/中学显示"即将推出"
- 数据库 schema 不变，已兼容三学段
- API `stage` 参数照常支持，但 MVP 只灌入幼稚园数据

---

## D2：登录方式

**决策：MVP 使用 Google 登录**

理由：
- Supabase Auth 原生支持 Google OAuth，几乎零成本接入
- 手机号 OTP 需要选择短信服务商，香港短信发送成本较高（约 HKD 0.3-0.5/条）
- MVP 阶段用户量不大，不需要为登录转化率过度优化
- 后续可以随时加入手机 OTP 作为补充登录方式

影响：
- `users` 表 `auth_provider` 记录为 `google`
- `users.phone` 在 MVP 阶段为空
- 登录弹层只展示 Google 登录按钮

---

## D3：距离排序

**决策：MVP 彻底不做距离排序**

理由：
- 需要用户授权定位 → 增加权限弹窗 → 影响首次体验
- 需要学校坐标数据质量保证 → 地址清洗工作量大
- 需要距离计算逻辑 → 增加后端复杂度
- 当前产品定位是"查学位"而非"找附近学校"

影响：
- 排序选项仅保留：推荐（有位优先）、更新时间最新、名称 A-Z
- `schools` 表 `lat`/`lng` 字段保留但 MVP 不填充
- 前端列表卡片不展示距离信息

---

## D4：后台推荐学校能力

**决策：MVP 不做推荐学校**

理由：
- 推荐学校涉及排序权重调整、推荐位管理、展示逻辑，属于运营功能
- MVP 核心目标是验证用户是否需要这个工具，不是优化运营效率
- 后续可以通过 `schools` 表加 `is_featured` 字段快速实现

影响：
- 后台不需要"推荐位"管理模块
- 列表页排序纯靠算法（有位优先 + 更新时间）

---

## D5：第三方数据源覆盖规则

**决策：第三方源只补空，不覆盖官方字段**

理由：
- 官方数据可信度最高
- 第三方数据可能过时或有误
- 一旦允许覆盖，数据溯源和 debug 复杂度急剧上升

影响：
- 同步写入规则：官方源 > 人工修改 > 第三方源
- 第三方源只在字段为空时填入
- `raw_data` 保留第三方原始数据用于后续人工审核

---

## D6：学费、Band、校网等字段展示

**决策：MVP 第一版详情页展示这些字段（如果有数据）**

理由：
- 这些字段在数据库已有，展示成本很低
- 对家长决策有参考价值
- "如有则展示"不增加数据清洗负担

影响：
- 详情页按学段动态展示相关字段
- 幼稚园：学费、全日/半日、是否参加计划
- 小学：校网、学校类型（MVP 数据暂为空）
- 中学：Band、学校类型（MVP 数据暂为空）

---

## D7：管理员认证方案

**决策：通过 `users` 表 `role` 字段标识管理员**

理由：
- 最简方案，不需要额外的权限系统
- MVP 管理员人数极少（1-2 人），手动在数据库设定 role 足够
- Supabase RLS（Row Level Security）可以直接用 role 做策略

实现：
- `users` 表增加 `role` 字段，默认值 `user`，管理员为 `admin`
- 后台接口中间件检查 `role = 'admin'`
- MVP 不做后台注册，手动在 DB 设置管理员角色

---

## D8：前端 UI 语言

**决策：MVP 前端界面使用繁体中文（粤语书面语）**

理由：
- 目标用户是香港家长
- 设计标准已明确"官方语言为粤语"
- 数据库中英文字段保留，但 UI 标签和交互文案统一用繁体中文

影响：
- 学校名称同时展示中英文（学校名称是数据字段，非 UI 文案）
- 按钮、标签、提示信息等为繁体中文
- MVP 不做国际化框架，硬编码即可

---

## D9：对比页和收藏页

**决策：MVP 需要独立的对比页和收藏页**

理由：
- 对比是核心功能，需要有表格形式的并排展示
- 收藏列表需要一个地方让用户集中查看
- 实现成本低（复用学校卡片组件 + 表格组件）

影响：
- 新增页面：`/compare`（对比页）、`/favorites`（收藏页）
- 需要在导航中增加入口
