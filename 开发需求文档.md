# HKSchoolPlace 开发需求文档

## 1. 文档目标

本文档用于将现有的产品想法整理为可执行的工程需求，作为 MVP 开发阶段的单一事实来源。

适用对象：
- 产品负责人
- 前端开发
- 后端开发
- 数据抓取开发
- 运营/后台管理

本文档目标：
- 明确 MVP 范围与边界
- 明确数据模型与业务规则
- 明确 API 合同
- 明确抓取与同步策略
- 明确开发顺序与验收标准

## 2. 产品定义

### 2.1 产品名称

- 产品名：HKSchoolPlace
- 面向地区：香港
- 目标用户：正在为子女寻找学位的家长

### 2.2 产品目标

帮助家长快速找到有学位的学校，并完成初步筛选、收藏与比较。

### 2.3 MVP 核心价值

- 提供按学段查看学校列表（MVP 第一版仅上线幼稚园）
- 提供 vacancy 信息和更新时间
- 提供基础筛选与排序
- 提供学校详情页
- 提供收藏与对比功能（含独立收藏页和对比页）
- 提供最小后台，用于学校展示管理和数据修正

### 2.4 MVP 不做

- 不做地图主视图
- 不做学校端后台
- 不做 lead 收集
- 不做在线咨询
- 不做复杂推荐算法
- 不做家长社区/评论
- 不做实时抓取后直接展示
- 不做距离排序（需要用户定位和坐标数据，开发成本高）
- 不做推荐学校/推荐位
- 不做国际化（MVP 界面统一繁体中文）

## 3. 关键产品原则

- 数据必须先入库、清洗、标准化，再对外提供
- vacancy 是列表页的第一优先级信息
- 浏览无需登录，收藏/对比时再触发登录
- 抓取失败不能覆盖旧数据
- 所有 vacancy 必须展示“最后更新时间”
- 筛选逻辑放在后端，不放在前端本地拼装

## 4. MVP 范围

#### 首页 `/`

功能：
- 选择学段：幼稚园 / 小学 / 中学

MVP 要求：
- 默认突出”查找有位学校”
- 允许不选地区直接搜索
- 小学和中学入口显示”即將推出”（MVP 仅上线幼稚园数据）

功能：

- 搜索学校名
- 使用筛选器缩小范围
- 使用排序查看结果
- 收藏学校
- 加入对比

列表卡片必须展示：
- 学校名称
- 学段
- 地区
- vacancy 状态
- vacancy 数量或描述
- 最后更新时间
- 学校类型标签
- 详情入口

#### 学校详情页 `/schools/:id`

功能：
- 查看学校基础信息
- 查看 vacancy 详情
- 查看基本属性
- 收藏
- 加入对比

必须展示：
- 学校中英文名
- 地址
- 电话
- 官网
- 学段
- 地区
- 学校类型
- 学费（如有）
- 校网（小学适用）
- Band（中学适用）
- vacancy 及更新时间

MVP 方案：
- 采用 Google 登录（Supabase Auth 原生支持，零成本接入）

触发点：
- 未登录用户点击收藏
- 未登录用户点击加入对比

登录交互：
- 弹出登录弹层（非跳转页）

#### 收藏页 `/favorites`

功能：
- 展示当前用户所有收藏的学校

- 需要登录才能访问

展示：

- 空态：显示"你還未收藏任何學校"，带一个跳转到列表页的按钮

#### 对比页 `/compare`

功能：
- 横向表格对比已选学校（2-5 间）
- 支持从对比列表中移除学校
- 需要登录才能访问

- 学校名称
- 地区
- 学校类型
- 上课时段（全日/半日）
- 学费

小学对比字段（后续）：

- 地区
- 校网

- P1 vacancy 状态

中学对比字段（后续）：

- 地区
- Band
- 学校类型
- S1 vacancy 状态
- vacancy 更新时间

UI 要求：

- 每列代表一间学校

## 5. 用户角色与权限

### 5.1 游客

- 可浏览首页、列表、详情
- 不可收藏
- 不可创建对比列表

### 5.2 登录用户

- 可收藏学校
- 可加入对比
- 可取消收藏
- 可维护自己的对比列表

### 5.3 管理员

- 可管理学校资料
- 可管理 vacancy 展示状态
- 可维护 alias 映射
- 可查看同步日志

管理员认证方案：
- `users` 表通过 `role` 字段标识，默认值 `user`，管理员为 `admin`
- 后台接口中间件统一检查 `role = 'admin'`
- MVP 不提供后台注册功能，管理员角色由数据库手动设置
- Supabase RLS 策略可直接使用 role 字段做权限控制

## 6. 业务规则

### 6.1 学段定义

- `kg`：幼稚园
- `primary`：小学
- `secondary`：中学

### 6.2 Vacancy 状态定义

- `available`：明确有学位
- `waiting`：候补 / waiting list / 不保证有位
- `full`：明确满额
- `unknown`：源数据缺失或无法判断

### 6.3 Vacancy 数量规则

- 若数据源提供数量，则写入 `count`
- 若数据源只提供文字状态，无明确数字，则 `count = null`
- 前端必须兼容只展示状态、不展示数值

### 6.4 更新时间规则

- `updated_at` 表示该条 vacancy 最后一次成功同步时间
- 若超过 72 小时未更新，则前端显示“数据可能较旧”
- 若抓取失败，不清空旧数据，只更新同步日志

### 6.5 对比规则

- 每个用户最多对比 5 间学校
- 已加入对比的学校不可重复加入

### 6.6 收藏规则

- 同一用户对同一学校只能收藏一次

## 7. 数据模型

以下为 MVP 建议数据表。正式开发时需配套 SQL migration。

### 7.1 `schools`

用途：学校主数据表

字段：
- `id` UUID PK
- `name_zh` text not null
- `name_en` text null
- `normalized_name` text not null
- `stage` text not null
- `district` text not null
- `address_zh` text null
- `address_en` text null
- `lat` numeric null
- `lng` numeric null
- `phone` text null
- `website` text null
- `school_type` text null
- `session_type` text null
- `school_net` text null
- `band` text null
- `tuition_fee` numeric null
- `curriculum` text null
- `is_scheme_participant` boolean null
- `is_active` boolean not null default true
- `created_at` timestamptz not null
- `updated_at` timestamptz not null

约束与索引：
- index on `stage`
- index on `district`
- index on `is_active`
- index on `normalized_name`

说明：
- `school_net` 主要用于小学
- `band` 主要用于中学
- `session_type` 主要用于幼稚园，如全日/半日

### 7.2 `vacancies`

用途：学校 vacancy 动态数据

字段：
- `id` UUID PK
- `school_id` UUID FK -> schools.id
- `grade` text not null
- `status` text not null
- `count` integer null
- `source` text not null
- `source_url` text null
- `effective_date` date null
- `created_at` timestamptz not null
- `updated_at` timestamptz not null
- `is_stale` boolean not null default false

约束与索引：
- index on `school_id`
- index on `status`
- index on `updated_at`
- unique on (`school_id`, `grade`, `source`)

### 7.3 `school_alias`

用途：解决同校不同名称问题

字段：
- `id` UUID PK
- `school_id` UUID FK -> schools.id
- `alias_name` text not null
- `normalized_alias_name` text not null
- `source` text not null
- `created_at` timestamptz not null

约束与索引：
- index on `school_id`
- index on `normalized_alias_name`
- unique on (`normalized_alias_name`, `source`)

### 7.4 `raw_data`

用途：保存抓取原始结果，用于 debug、回滚、比对

字段：
- `id` UUID PK
- `source` text not null
- `source_type` text not null
- `payload` jsonb not null
- `fetched_at` timestamptz not null
- `status` text not null
- `error_message` text null

索引：
- index on `source`
- index on `fetched_at`

### 7.5 `sync_logs`

用途：记录同步过程

字段：
- `id` UUID PK
- `source` text not null
- `run_type` text not null
- `status` text not null
- `records_fetched` integer not null default 0
- `records_parsed` integer not null default 0
- `records_matched` integer not null default 0
- `records_updated` integer not null default 0
- `message` text null
- `started_at` timestamptz not null
- `finished_at` timestamptz null

### 7.6 `users`

字段：
- `id` UUID PK
- `auth_provider` text not null
- `display_name` text null
- `phone` text null
- `email` text null
- `role` text not null default 'user'
- `created_at` timestamptz not null

说明：
- `role` 取值：`user`（默认）、`admin`
- MVP 管理员角色在数据库手动设置
- `display_name` 从 Google OAuth 获取

### 7.7 `favorites`

字段：
- `id` UUID PK
- `user_id` UUID FK -> users.id
- `school_id` UUID FK -> schools.id
- `created_at` timestamptz not null

约束：
- unique on (`user_id`, `school_id`)

### 7.8 `comparisons`

字段：
- `id` UUID PK
- `user_id` UUID FK -> users.id
- `school_id` UUID FK -> schools.id
- `created_at` timestamptz not null

约束：
- unique on (`user_id`, `school_id`)

### 7.9 `unmatched_records`

用途：保存抓取过程中无法匹配到学校的记录，等待人工处理

字段：
- `id` UUID PK
- `source` text not null
- `raw_name` text not null
- `normalized_name` text not null
- `stage` text null
- `district` text null
- `grade` text null
- `suggested_school_id` UUID null FK -> schools.id
- `status` text not null default 'pending'
- `resolved_school_id` UUID null FK -> schools.id
- `resolved_at` timestamptz null
- `resolved_by` UUID null FK -> users.id
- `created_at` timestamptz not null

约束与索引：
- index on `status`
- index on `source`
- index on `created_at`

说明：
- `status` 取值：`pending`（待处理）、`resolved`（已绑定学校）、`ignored`（已忽略）
- `suggested_school_id` 由系统自动推荐的候选学校
- `resolved_school_id` 由管理员手动确认的学校
- 管理员处理后，系统可自动创建对应 alias

## 8. 数据源与抓取策略

### 8.1 数据源分层

建议将数据分为两类：

#### 学校主数据源

用途：学校名称、地址、地区、类型、联系方式等

候选源：
- 香港政府公开数据 `data.gov.hk`
- 教育局公开学校资料
- 其他稳定学校目录网站

#### Vacancy 动态数据源

用途：学位空缺、候补、满额状态

候选源：
- EDB 幼稚园 vacancy 页面
- 其他具备 vacancy 信息的网站

### 8.2 数据源优先级

建议规则：
- 学校主数据以官方源优先
- vacancy 以官方 vacancy 源优先
- 第三方源只用于补充缺失字段，不直接覆盖官方字段

### 8.3 抓取频率

- vacancy：每日 1 到 2 次
- 学校主数据：每周 1 次
- alias 自动识别：每次同步后执行

### 8.4 抓取流程

1. 拉取源数据
2. 原样写入 `raw_data`
3. 解析为标准结构
4. 标准化学校名称
5. 根据 alias 和 normalized name 匹配 `school_id`
6. 将成功匹配结果更新到 `schools` 或 `vacancies`
7. 记录 `sync_logs`

### 8.5 名称标准化规则

MVP 版本至少做以下标准化：
- 去除首尾空格
- 多空格合并为单空格
- 统一英文大小写
- 去除常见标点
- 统一“幼稚园 / 幼儿园 / kindergarten / kg”等常见后缀
- 将繁简差异收敛到固定形式

### 8.6 匹配规则

匹配顺序：

1. 精确匹配 `school_alias.normalized_alias_name`
2. 精确匹配 `schools.normalized_name`
3. 基于名称和地区的高置信匹配
4. 未匹配的记录进入人工处理池

MVP 要求：
- 不允许低置信度自动写入错误学校
- 匹配不到时可跳过更新，并写入日志

### 8.7 失败处理

- 抓取失败：保留历史业务数据
- 解析失败：仅跳过失败记录，不影响其他记录
- 匹配失败：记录到日志并等待后台人工校正

## 11. API 合同

接口前缀建议：
- 用户端：`/api/v1`
- 后台端：`/api/admin/v1`

### 11.1 `GET /api/v1/schools`

用途：获取学校列表

Query 参数：
- `stage` string required
- `district` string optional
- `keyword` string optional
- `has_vacancy` boolean optional
- `school_type` string optional
- `session_type` string optional
- `school_net` string optional
- `band` string optional
- `page` integer default 1
- `page_size` integer default 20 max 50
- `sort` string default `recommended`

返回示例：

```json
{
  "data": [
    {
      "id": "uuid",
      "name_zh": "ABC幼稚园",
      "name_en": "ABC Kindergarten",
      "stage": "kg",
      "district": "九龙城",
      "school_type": "非牟利",
      "session_type": "全日",
      "vacancy": {
        "grade": "K1",
        "status": "available",
        "count": 8,
        "updated_at": "2026-03-18T10:00:00+08:00",
        "is_stale": false
      }
    }
  ],
  "pagination": {
    "page": 1,
    "page_size": 20,
    "total": 120
  }
}
```

### 11.2 `GET /api/v1/schools/:id`

用途：获取学校详情

返回字段：
- 学校基础信息
- 当前展示的 vacancy 列表
- 收藏状态
- 是否已加入对比

### 11.3 `GET /api/v1/filters`

用途：获取筛选项配置

返回内容：
- 地区列表
- 学校类型列表
- 学段对应筛选项
- 校网列表
- Band 列表

### 11.4 `POST /api/v1/favorites`

请求体：

```json
{
  "school_id": "uuid"
}
```

行为：
- 未登录返回 401
- 已存在则返回成功且不重复创建

### 11.5 `DELETE /api/v1/favorites/:school_id`

行为：
- 删除用户自己的收藏

### 11.6 `GET /api/v1/favorites`

用途：获取我的收藏列表

### 11.7 `POST /api/v1/comparisons`

请求体：

```json
{
  "school_id": "uuid"
}
```

行为：
- 未登录返回 401
- 超过 5 条返回 422（业务校验失败）

### 11.8 `DELETE /api/v1/comparisons/:school_id`

用途：移除对比学校

### 11.9 `GET /api/v1/comparisons`

用途：获取当前用户对比列表

### 11.10 后台接口

MVP 至少需要：
- `GET /api/admin/v1/schools`
- `PATCH /api/admin/v1/schools/:id`
- `GET /api/admin/v1/vacancies`
- `PATCH /api/admin/v1/vacancies/:id`
- `GET /api/admin/v1/aliases`
- `POST /api/admin/v1/aliases`
- `PATCH /api/admin/v1/aliases/:id`
- `GET /api/admin/v1/sync-logs`

## 13. 技术方案建议

### 13.1 推荐栈

- 前端：Next.js
- UI：Tailwind CSS
- 后端：Next.js Route Handlers
- 数据库：Supabase Postgres
- 认证：Supabase Auth
- 抓取任务：Node.js + Cron

### 13.2 原因

- 同一套仓库完成前后端，降低 MVP 开发成本
- Supabase 适合快速建立数据库、认证和后台能力
- Route Handlers 足够承载 MVP API

### 13.3 已确定的技术决策

- 使用 App Router（项目已初始化为 App Router）
- 登录方式：MVP 使用 Google 登录（Supabase Auth）
- 抓取任务：Node.js 脚本 + Cron，部署在同一服务器或 Vercel Cron Jobs
- 管理后台：内嵌在主站 `/admin` 路由下，共享同一套仓库

## 14. 多语言策略

### 14.1 MVP 语言方案

- 前端 UI 界面：繁体中文（粤语书面语）
- 学校数据：中英文字段同时存储和展示（学校名称展示中文为主，英文为辅）
- 前端不做国际化框架（不引入 i18n 库），文案直接硬编码繁体中文
- 后续如需多语言再引入 `next-intl` 或类似方案

### 14.2 影响

- 所有按钮、标签、提示信息、空态文案使用繁体中文
- 系统枚举的中文显示值使用繁体写法（如"學校"非"学校"）
- 数据库字段名和 API 参数名保持英文

## 14. 非功能要求

### 14.1 性能

- 列表页首屏接口返回时间目标：1 秒内
- 接口必须分页
- 高频查询可做缓存

### 14.2 SEO

- 首页、列表页、详情页应支持基础 SEO
- 详情页需有可读 URL 或 canonical 方案

SEO 实施方案：

URL 结构：
- 首页：`/`
- 列表页：`/schools?stage=kg&district=九龍城`
- 详情页：`/schools/[id]`（UUID，并通过 canonical 标签标准化）
- 收藏页：`/favorites`（不需要 SEO，需登录）
- 对比页：`/compare`（不需要 SEO，需登录）

Meta Tags（每页必须配置）：
- `<title>`：动态生成，如"幼稚園學位空缺 - 九龍城 | 學位通"
- `<meta name="description">`：动态生成，含学段、地区、vacancy 概况
- `<meta property="og:title">` / `og:description` / `og:image`：社交分享用

Sitemap：
- 自动生成 `/sitemap.xml`
- 包含所有 `is_active = true` 的学校详情页
- 每日更新

其他：
- 使用 Next.js App Router 的 `generateMetadata` 实现动态 meta
- 列表页和详情页使用 SSR 或 SSG，确保搜索引擎可抓取
- 管理后台页面添加 `<meta name="robots" content="noindex">`

### 14.3 日志与监控

- 所有同步任务必须写日志
- API 需记录基础错误日志
- 管理员能看到最近同步是否失败

### 14.4 安全

- 后台接口必须鉴权
- 用户接口必须校验 user_id
- 不信任前端传入的收藏/对比归属
